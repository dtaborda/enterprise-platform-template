# Production Deployment

## Purpose

This guide walks through every step required to deploy the Enterprise Platform to a production environment. It covers the full infrastructure stack: Supabase Cloud, Vercel, Sentry, and Resend.

## Scope

- **Included**: Supabase production project, Vercel deployment, Sentry error tracking, Resend email, post-deploy verification, troubleshooting
- **Excluded**: Custom domains/DNS configuration, CDN tuning, database performance optimization

---

## Overview: The Production Stack

| Layer | Service | Purpose |
|-------|---------|---------|
| Hosting | [Vercel](https://vercel.com) | Next.js application hosting, edge functions, CI/CD |
| Database + Auth | [Supabase Cloud](https://supabase.com) | PostgreSQL, Row Level Security, Auth, Storage |
| Error Tracking | [Sentry](https://sentry.io) | Error capture, performance monitoring, alerting |
| Transactional Email | [Resend](https://resend.com) | Welcome emails, password resets, notifications |

> **đźš¨ Production vs dev Supabase:** Always create a SEPARATE Supabase project for production. Never point production to a shared dev project. Migrations run against real user data â€” there is no undo.

## Step 1: Supabase Cloud Setup

### 1.1 Create the Production Project

1. Log in to [supabase.com](https://supabase.com).
2. Click **New Project** and set a name like `my-app-production`.
3. Choose the region closest to your users.
4. Set a strong database password â€” save it in your password manager.
5. Note the **Project Reference ID** from the project URL.

### 1.2 Push Migrations

**Link CLI and push migrations**

```bash
supabase link --project-ref <PROD_REF>
supabase db push
```

> **âš ď¸Ź Review before pushing:** Run `supabase db diff` first to preview the SQL that will be applied. In production, there is no `db reset` â€” changes are permanent and incremental.

### 1.3 Configure Auth

In **Dashboard â†’ Authentication â†’ URL Configuration**:

| Setting | Value |
|---------|-------|
| **Site URL** | Your production domain (e.g., `https://your-app.com`) |
| **Redirect URLs** | `https://your-app.com/**` plus any preview URLs |

Enable email confirmations in **Authentication â†’ Providers â†’ Email** if your product requires verified email addresses.

### 1.4 Verify RLS

```sql
SELECT schemaname, tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY tablename;
```

Every table must show `rowsecurity = true`. If any table is unprotected, do not launch.

### 1.5 Create Storage Buckets

```sql
INSERT INTO storage.buckets (id, name, public) VALUES
  ('avatars', 'avatars', true),
  ('documents', 'documents', false),
  ('images', 'images', true);
```

## Step 2: GitHub Secrets

The CI workflow reads secrets from **GitHub repository Settings â†’ Secrets and variables â†’ Actions**. Configure all of these before your first push to `main`.

### 2.1 Add Secrets

1. Go to your GitHub repository â†’ **Settings â†’ Secrets and variables â†’ Actions**.
2. Click **New repository secret**.
3. Add each secret from the table below.

| Secret | Required | Where to get it | Used by |
|--------|----------|----------------|---------|
| `NEXT_PUBLIC_SUPABASE_URL` | Yes | Supabase Dashboard â†’ Settings â†’ API | CI build |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Yes | Supabase Dashboard â†’ Settings â†’ API | CI build |
| `NEXT_PUBLIC_APP_URL` | Yes | Your production URL (e.g. `https://myapp.vercel.app`) | CI build |
| `NEXT_PUBLIC_APP_NAME` | Yes | Your app name | CI build |
| `NEXT_PUBLIC_APP_ENV` | Yes | Set to `production` | CI build |
| `SUPABASE_ACCESS_TOKEN` | Yes | supabase.com â†’ Account â†’ Access Tokens | Migration deploy |
| `SUPABASE_PROJECT_ID` | Yes | Supabase Dashboard URL â†’ project ref | Migration deploy |
| `SUPABASE_DB_PASSWORD` | Yes | The password you set when creating the project | Migration deploy |
| `NEXT_PUBLIC_SENTRY_DSN` | Optional | Sentry â†’ Project â†’ Settings â†’ Client Keys | CI build + Sentry |
| `SENTRY_AUTH_TOKEN` | Optional | sentry.io â†’ Settings â†’ Auth Tokens | Source map upload |
| `SENTRY_ORG` | Optional | Sentry organization slug | Source map upload |
| `SENTRY_PROJECT` | Optional | Sentry project slug | Source map upload |

> **â„ąď¸Ź Sentry secrets are optional:** If `SENTRY_AUTH_TOKEN` is absent, source maps are not uploaded but the build still succeeds (`silent: true` is set). Sentry event capture requires only `NEXT_PUBLIC_SENTRY_DSN` at runtime.

## Step 3: Vercel Setup

### 3.1 Import Repository

1. Go to [vercel.com](https://vercel.com) â†’ **Add New Project**.
2. Import your repository from GitHub/GitLab/Bitbucket.
3. Set **Framework Preset** to **Next.js**.

### 3.2 Build Settings

| Setting | Value |
|---------|-------|
| **Build Command** | `pnpm build` |
| **Install Command** | `pnpm install` |
| **Output Directory** | `.next` (default) |
| **Root Directory** | `ui/` (if deploying only the web app from a monorepo root) |

> **â„ąď¸Ź Monorepo root directory:** If Vercel auto-detects the monorepo root and runs `pnpm build` from there, it will build all packages. Setting the Root Directory to `ui/` scopes the deployment to the Next.js app only. Check your Turborepo `build` pipeline before deciding.

### 3.3 Vercel Environment Variables

Add every variable in **Project â†’ Settings â†’ Environment Variables**, selecting the appropriate scope for each.

**Steps:**

1. Go to Vercel â†’ Your Project â†’ **Settings â†’ Environment Variables**.
2. Click **Add New**, enter the variable name and value.
3. Select the appropriate scope: **Production**, **Preview**, and/or **Development**.
4. For secrets like `SUPABASE_SERVICE_ROLE_KEY`, enable the **Sensitive** toggle â€” sensitive variables are write-only after saving.

| Variable | Scope | Value |
|----------|-------|-------|
| `NEXT_PUBLIC_SUPABASE_URL` | Production, Preview | `https://<ref>.supabase.co` |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Production, Preview | From Supabase API settings |
| `SUPABASE_SERVICE_ROLE_KEY` | Production, Preview | From Supabase API settings **(SECRET)** |
| `NEXT_PUBLIC_APP_URL` | Production | Your production URL |
| `NEXT_PUBLIC_APP_NAME` | All | Your app display name |
| `NEXT_PUBLIC_APP_ENV` | Production: `production` / Preview: `preview` | Per environment |
| `NEXT_PUBLIC_SENTRY_DSN` | All | From Sentry client keys |
| `SENTRY_AUTH_TOKEN` | All | From Sentry auth tokens **(SECRET)** |
| `SENTRY_ORG` | All | Your Sentry org slug |
| `SENTRY_PROJECT` | All | Your Sentry project slug |
| `RESEND_API_KEY` | Production | From resend.com API keys **(SECRET)** |
| `EMAIL_FROM` | Production | Your verified sender email |

> **đźš¨ Sensitive variables:** Mark `SUPABASE_SERVICE_ROLE_KEY`, `SENTRY_AUTH_TOKEN`, and `RESEND_API_KEY` as **Sensitive** in Vercel. Sensitive variables are write-only â€” they cannot be viewed after saving.

## Step 4: Sentry Setup (Optional)

Sentry is **pre-instrumented** in this template â€” the SDK is already installed, configured, and wired into Next.js. You only need to create a Sentry project and set environment variables.

> **No installation needed.** `@sentry/nextjs` is already in `ui/package.json`. Instrumentation files (`instrumentation.ts`, `instrumentation-client.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`) are already created and configured.

### 4.1 Create a Sentry Project

1. Go to [sentry.io](https://sentry.io) â†’ **Projects â†’ Create Project**.
2. Select **Next.js** as the platform.
3. Copy the **DSN** from the project settings page.

### 4.2 Set Environment Variables

Set these in both Vercel (Step 3.3) and GitHub Secrets (Step 2.1):

| Variable | Where to find it |
|----------|-----------------|
| `NEXT_PUBLIC_SENTRY_DSN` | Sentry â†’ Project â†’ Settings â†’ Client Keys |
| `SENTRY_AUTH_TOKEN` | sentry.io â†’ Settings â†’ Auth Tokens (create a new token) |
| `SENTRY_ORG` | Your Sentry organization slug (from the URL) |
| `SENTRY_PROJECT` | Your Sentry project slug |

### 4.3 What Is Already Configured

| Feature | Status |
|---------|--------|
| SDK installed | âś… `@sentry/nextjs ^10.48.0` |
| Source maps auto-uploaded | âś… During Vercel builds when `SENTRY_AUTH_TOKEN` is set |
| Source maps deleted after upload | âś… Not exposed to browsers |
| `NEXT_REDIRECT` and `NEXT_NOT_FOUND` filtered | âś… Dropped in `beforeSend` â€” not real errors |
| PII scrubbed from sensitive fields | âś… `password`, `token`, `secret`, `card`, `access_token`, `service_role`, `authorization` |
| Replay sessions disabled until privacy review | âś… `replaysSessionSampleRate: 0.0` |
| On-error replay enabled | âś… `replaysOnErrorSampleRate: 1.0` |
| Sentry disabled in local dev | âś… `enabled: process.env.NODE_ENV === "production"` |
| Build succeeds without auth token | âś… `silent: true` |
| Enterprise tag taxonomy | âś… `enterprise.*` tag prefix for all custom tags |

### 4.4 Source Maps

Source maps are uploaded automatically during Vercel builds when `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, and `SENTRY_PROJECT` are configured. Source map files are deleted from the deployment after upload â€” they are never exposed in the browser.

## Step 5: Resend Setup (Optional)

Resend handles transactional emails: welcome messages, password resets, and notifications.

### 5.1 Create Account and API Key

1. Go to [resend.com](https://resend.com) and sign up.
2. Navigate to **API Keys â†’ Create API Key**.
3. Copy the key â€” it is shown only once.

### 5.2 Verify Your Domain

For production deliverability, verify your sending domain in **Resend â†’ Domains â†’ Add Domain**. Follow the DNS verification instructions. Without domain verification, emails may be sent from `onboarding@resend.dev` and may end up in spam.

### 5.3 Add to Vercel

Add to Vercel environment variables:

```
RESEND_API_KEY=re_...
EMAIL_FROM=noreply@yourdomain.com
```

## Step 6: CI/CD and Automated Migration Deployment

The GitHub Actions CI workflow (`ci.yml`) runs on every pull request to `main` and every push to `main`.

### Pipeline Overview

| Step | When | What it does |
|------|------|--------------|
| Typecheck | Always | `pnpm typecheck` â€” TypeScript strict check |
| Lint | Always | `pnpm lint` â€” Biome lint rules |
| Test | Always | `pnpm test` â€” Vitest unit tests |
| Build | Always | `pnpm build` â€” Next.js production build |
| Deploy Migrations | Push to `main` only | `supabase db push` â€” applies pending migrations |

### Automated Migration Deployment

Migrations run automatically **after a successful build** on every push to `main`. This means:

- The build must pass before migrations are applied â€” broken code never reaches the database.
- Migrations are applied in order using the Supabase CLI.
- Required secrets: `SUPABASE_ACCESS_TOKEN`, `SUPABASE_PROJECT_ID`, `SUPABASE_DB_PASSWORD`.

> **âš ď¸Ź Migrations are irreversible in production.** Always test migrations in a staging environment first. Use `supabase db diff` locally to preview what will be applied.

### Required Secrets for Migration Deployment

| Secret | Purpose |
|--------|---------|
| `SUPABASE_ACCESS_TOKEN` | Authenticates the Supabase CLI |
| `SUPABASE_PROJECT_ID` | Identifies the production project to link to |
| `SUPABASE_DB_PASSWORD` | Required by `supabase db push` |

## Step 7: Deploy

### 7.1 Push to Main

**Trigger production deploy**

```bash

```

Vercel auto-deploys every push to `main`. Monitor the build in the Vercel dashboard.

The CI pipeline will also run: typecheck â†’ lint â†’ test â†’ build â†’ deploy migrations.

### 7.2 Verify Production URL

After the deployment completes, visit your production URL and confirm:

- The landing page loads.
- `/sign-in` loads and accepts credentials.
- Auth redirects work (forgot password, sign-up confirmation).

## Post-Deploy Checklist

Run through this list after every first-time production deployment:

- [ ] Visit production URL â€” no 500 errors on landing page
- [ ] Sign in with a real user account
- [ ] Sign up a new user â€” verify tenant and profile are created
- [ ] Trigger a password reset â€” verify email arrives
- [ ] Access `/dashboard` â€” verify all data queries return correct tenant-scoped data
- [ ] Verify RLS: check that a user cannot access another tenant's data
- [ ] Open Sentry dashboard â€” confirm events are appearing after a test error
- [ ] Check Resend dashboard â€” confirm emails are being sent and delivered
- [ ] Run `supabase db push` â€” confirm no pending migrations

## Troubleshooting

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Build fails: "Cannot find module" | Missing workspace dependency | Check `turbo.json` build pipeline and workspace `package.json` |
| Auth redirect goes to wrong URL | `NEXT_PUBLIC_APP_URL` misconfigured | Set to exact production domain, including `https://` |
| RLS blocking all queries | Missing `tenant_id` in JWT `app_metadata` | Verify `handle_new_user()` trigger exists and runs on sign-up |
| "Localhost URL rejected in production" startup error | `NEXT_PUBLIC_APP_URL` set to `localhost` | Set to real production domain |
| Sentry not receiving events | Wrong DSN or `NEXT_PUBLIC_SENTRY_DSN` not set | Copy exact DSN from Sentry project settings |
| Source maps not uploaded | `SENTRY_AUTH_TOKEN` missing | Add token to Vercel env vars and GitHub Secrets |
| Emails not sending | `RESEND_API_KEY` missing or domain not verified | Add API key to Vercel env vars and verify sender domain |
| Storage uploads failing | Bucket not created or RLS policy missing | Create buckets manually and check `storage.objects` RLS policies |
| Vercel build uses wrong Node version | Missing `.nvmrc` or engines field | Add `"engines": { "node": ">=20" }` to root `package.json` |
| Migration deploy fails | `SUPABASE_ACCESS_TOKEN` or `SUPABASE_PROJECT_ID` missing | Add secrets to GitHub repo settings |
