# Production Deployment

## Purpose

This guide walks through every step required to deploy the Enterprise Platform to a production environment. It covers the full infrastructure stack: Supabase Cloud, Vercel, Sentry, and Resend.

## Scope

- **Included**: Supabase production project, Vercel deployment, Sentry error tracking, Resend email, post-deploy verification, troubleshooting
- **Excluded**: Custom domains/DNS configuration, CDN tuning, database performance optimization

## Before You Continue

This guide is for the **production-ready** path.

If you only want the fastest hosted MVP, stop here and use [MVP Cloud Deploy](./mvp-cloud-deploy.mdx) instead. That path requires **no GitHub Secrets** and no CI automation.

### When do I need GitHub Secrets?

| Path | GitHub Secrets needed? |
|------|----------------------|
| **Local development** | No |
| **MVP cloud deploy** (manual `supabase db push`) | No |
| **Production automation** (CI-driven migration deploy) | Yes |

### Use this guide when you want:

- automated database deploys from a separate CI workflow
- Sentry already wired for traceability
- optional Resend email ready for production
- GitHub Secrets and Vercel env vars fully configured

## Setup Model for This Template

This template intentionally separates deployment maturity into three levels:

### Required for a hosted MVP

- Supabase cloud project
- Manual `supabase db push`
- Minimum Vercel environment variables

### Required for production automation

- GitHub Secrets for CI build and migration deployment
- Repository variable: `ENABLE_SUPABASE_MIGRATIONS=true`
- Separate migration deployment workflow enabled on `main`

### Optional for observability and email

- Sentry DSN and source map upload credentials
- Resend API key and verified sender domain

This means the template is safe by default: a new team can deploy an MVP without first configuring GitHub automation.

## Overview: The Production Stack

| Layer | Service | Purpose |
|-------|---------|---------|
| Hosting | [Vercel](https://vercel.com) | Next.js application hosting, edge functions, CI/CD |
| Database + Auth | [Supabase Cloud](https://supabase.com) | PostgreSQL, Row Level Security, Auth, Storage |
| Error Tracking | [Sentry](https://sentry.io) | Error capture, performance monitoring, alerting |
| Transactional Email | [Resend](https://resend.com) | Welcome emails, password resets, notifications |

> **🚨 Production vs dev Supabase:** Always create a SEPARATE Supabase project for production. Never point production to a shared dev project. Migrations run against real user data — there is no undo.

## Step 1: Supabase Cloud Setup

### 1.1 Create the Production Project

1. Log in to [supabase.com](https://supabase.com).
2. Click **New Project** and set a name like `my-app-production`.
3. Choose the region closest to your users.
4. Set a strong database password — save it in your password manager.
5. Note the **Project Reference ID** from the project URL.

### 1.2 Push Migrations

**Link CLI and push migrations**

```bash
supabase link --project-ref <PROD_REF>
supabase db push
```

> **⚠️ Review before pushing:** Run `supabase db diff` first to preview the SQL that will be applied. In production, there is no `db reset` — changes are permanent and incremental.

### 1.3 Configure Auth

In **Dashboard → Authentication → URL Configuration**:

| Setting | Value |
|---------|-------|
| **Site URL** | Your production domain (e.g., `https://your-app.com`) |
| **Redirect URLs** | `https://your-app.com/**` plus any preview URLs |

Enable email confirmations in **Authentication → Providers → Email** if your product requires verified email addresses.

### 1.4 Verify RLS

```sql
SELECT schemaname, tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY tablename;
```

Every table must show `rowsecurity = true`. If any table is unprotected, do not launch.

### 1.5 Create Storage Buckets

```sql
INSERT INTO storage.buckets (id, name, public) VALUES
  ('avatars', 'avatars', true),
  ('documents', 'documents', false),
  ('images', 'images', true);
```

## Step 2: GitHub Secrets and Variables

GitHub Secrets and Variables are configured in **Settings → Secrets and variables → Actions**.

> **ℹ️ Are all of these required?** No.
>
> - **Required for CI build:** the `NEXT_PUBLIC_*` values used by `pnpm build`
> - **Required for automated migration deploy:** `SUPABASE_ACCESS_TOKEN`, `SUPABASE_PROJECT_ID`, `SUPABASE_DB_PASSWORD` + the `ENABLE_SUPABASE_MIGRATIONS` repository variable
> - **Optional for observability:** the Sentry secrets
>
> If you are following the **MVP path** and deploying manually, you do **not** need any GitHub Secrets. The CI workflow runs typecheck, lint, test, and build — it does not deploy migrations.

### 2.1 Secrets for CI Build

1. Go to your GitHub repository → **Settings → Secrets and variables → Actions → Secrets**.
2. Click **New repository secret**.
3. Add each secret from the table below.

| Secret | Required for | Where to get it |
|--------|-------------|----------------|
| `NEXT_PUBLIC_SUPABASE_URL` | CI build | Supabase Dashboard → Settings → API |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | CI build | Supabase Dashboard → Settings → API |
| `NEXT_PUBLIC_APP_URL` | CI build | Your production URL (e.g. `https://myapp.vercel.app`) |
| `NEXT_PUBLIC_APP_NAME` | CI build | Your app name |
| `NEXT_PUBLIC_APP_ENV` | CI build | Set to `production` |
| `NEXT_PUBLIC_SENTRY_DSN` | Optional — Sentry | Sentry → Project → Settings → Client Keys |
| `SENTRY_AUTH_TOKEN` | Optional — source maps | sentry.io → Settings → Auth Tokens |
| `SENTRY_ORG` | Optional — source maps | Sentry organization slug |
| `SENTRY_PROJECT` | Optional — source maps | Sentry project slug |

> **ℹ️ Sentry secrets are optional:** If `SENTRY_AUTH_TOKEN` is absent, source maps are not uploaded but the build still succeeds (`silent: true` is set). Sentry event capture requires only `NEXT_PUBLIC_SENTRY_DSN` at runtime.

### 2.2 Secrets for Migration Deployment (Optional)

These secrets are only required if you enable the separate migration deployment workflow.

| Secret | Where to get it |
|--------|----------------|
| `SUPABASE_ACCESS_TOKEN` | supabase.com → Account → Access Tokens |
| `SUPABASE_PROJECT_ID` | Supabase Dashboard URL → project ref |
| `SUPABASE_DB_PASSWORD` | The password you set when creating the project |

### 2.3 Repository Variable for Migration Deployment

To enable automated migration deployment, add a **repository variable** (not a secret):

1. Go to **Settings → Secrets and variables → Actions → Variables**.
2. Click **New repository variable**.
3. Name: `ENABLE_SUPABASE_MIGRATIONS`, Value: `true`.

Without this variable, the migration deployment workflow is skipped entirely — safe for template forks that have not configured Supabase secrets yet.

### Why migration deployment is separate from CI

The main `ci.yml` workflow is intentionally limited to quality gates:

- `pnpm typecheck`
- `pnpm lint`
- `pnpm test`
- `pnpm build`

For a template, this is important because most adopters do **not** have production Supabase credentials configured on day one. If migration deployment lived inside the main CI workflow, a template fork could fail immediately after merge even though the code itself is healthy.

By keeping migration deployment in a separate workflow:

- local adopters are not blocked
- MVP cloud adopters can deploy manually
- production teams can opt in when they are ready

### Minimum GitHub Secrets for Full Automation

If you want CI build + automated migration deploy, the minimum set is:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `NEXT_PUBLIC_APP_URL`
- `NEXT_PUBLIC_APP_NAME`
- `NEXT_PUBLIC_APP_ENV`
- `SUPABASE_ACCESS_TOKEN`
- `SUPABASE_PROJECT_ID`
- `SUPABASE_DB_PASSWORD`
- Repository variable: `ENABLE_SUPABASE_MIGRATIONS=true`

Add the Sentry secrets only if you want observability and source maps.

## Step 3: Vercel Setup

### 3.1 Import Repository

1. Go to [vercel.com](https://vercel.com) → **Add New Project**.
2. Import your repository from GitHub/GitLab/Bitbucket.
3. Set **Framework Preset** to **Next.js**.

### 3.2 Build Settings

| Setting | Value |
|---------|-------|
| **Build Command** | `pnpm build` |
| **Install Command** | `pnpm install` |
| **Output Directory** | `.next` (default) |
| **Root Directory** | `ui/` (if deploying only the web app from a monorepo root) |

> **ℹ️ Monorepo root directory:** If Vercel auto-detects the monorepo root and runs `pnpm build` from there, it will build all packages. Setting the Root Directory to `ui/` scopes the deployment to the Next.js app only. Check your Turborepo `build` pipeline before deciding.

### 3.3 Vercel Environment Variables

Add every variable in **Project → Settings → Environment Variables**, selecting the appropriate scope for each.

**Steps:**

1. Go to Vercel → Your Project → **Settings → Environment Variables**.
2. Click **Add New**, enter the variable name and value.
3. Select the appropriate scope: **Production**, **Preview**, and/or **Development**.
4. For secrets like `SUPABASE_SERVICE_ROLE_KEY`, enable the **Sensitive** toggle — sensitive variables are write-only after saving.

| Variable | Scope | Value |
|----------|-------|-------|
| `NEXT_PUBLIC_SUPABASE_URL` | Production, Preview | `https://<ref>.supabase.co` |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Production, Preview | From Supabase API settings |
| `SUPABASE_SERVICE_ROLE_KEY` | Production, Preview | From Supabase API settings **(SECRET)** |
| `NEXT_PUBLIC_APP_URL` | Production | Your production URL |
| `NEXT_PUBLIC_APP_NAME` | All | Your app display name |
| `NEXT_PUBLIC_APP_ENV` | Production: `production` / Preview: `preview` | Per environment |
| `NEXT_PUBLIC_SENTRY_DSN` | All | From Sentry client keys |
| `SENTRY_AUTH_TOKEN` | All | From Sentry auth tokens **(SECRET)** |
| `SENTRY_ORG` | All | Your Sentry org slug |
| `SENTRY_PROJECT` | All | Your Sentry project slug |
| `RESEND_API_KEY` | Production | From resend.com API keys **(SECRET)** |
| `EMAIL_FROM` | Production | Your verified sender email |

> **🚨 Sensitive variables:** Mark `SUPABASE_SERVICE_ROLE_KEY`, `SENTRY_AUTH_TOKEN`, and `RESEND_API_KEY` as **Sensitive** in Vercel. Sensitive variables are write-only — they cannot be viewed after saving.

### Minimum Vercel Variables for a Production MVP

If you only want a working production deployment without optional integrations, the minimum required set is:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `NEXT_PUBLIC_APP_URL`
- `NEXT_PUBLIC_APP_NAME`
- `NEXT_PUBLIC_APP_ENV`

Everything else in the table is optional or recommended for production hardening.

## Step 4: Sentry Setup (Optional)

Sentry is **pre-instrumented** in this template — the SDK is already installed, configured, and wired into Next.js. You only need to create a Sentry project and set environment variables.

> **No installation needed.** `@sentry/nextjs` is already in `ui/package.json`. Instrumentation files (`instrumentation.ts`, `instrumentation-client.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`) are already created and configured.

### 4.1 Create a Sentry Project

1. Go to [sentry.io](https://sentry.io) → **Projects → Create Project**.
2. Select **Next.js** as the platform.
3. Copy the **DSN** from the project settings page.

### 4.2 Set Environment Variables

Set these in both Vercel (Step 3.3) and GitHub Secrets (Step 2.1):

| Variable | Where to find it |
|----------|-----------------|
| `NEXT_PUBLIC_SENTRY_DSN` | Sentry → Project → Settings → Client Keys |
| `SENTRY_AUTH_TOKEN` | sentry.io → Settings → Auth Tokens (create a new token) |
| `SENTRY_ORG` | Your Sentry organization slug (from the URL) |
| `SENTRY_PROJECT` | Your Sentry project slug |

### Release automation note

This template uses `release-please` for versioning and changelog generation. It manages:

- release PRs
- changelog updates
- git tags / releases

It does **not** manage labels for ordinary feature or fix PRs. If you expect labels to change automatically on normal PRs, that requires a separate workflow and is intentionally not part of the template by default.

### 4.3 What Is Already Configured

| Feature | Status |
|---------|--------|
| SDK installed | ✅ `@sentry/nextjs ^10.48.0` |
| Source maps auto-uploaded | ✅ During Vercel builds when `SENTRY_AUTH_TOKEN` is set |
| Source maps deleted after upload | ✅ Not exposed to browsers |
| `NEXT_REDIRECT` and `NEXT_NOT_FOUND` filtered | ✅ Dropped in `beforeSend` — not real errors |
| PII scrubbed from sensitive fields | ✅ `password`, `token`, `secret`, `card`, `access_token`, `service_role`, `authorization` |
| Replay sessions disabled until privacy review | ✅ `replaysSessionSampleRate: 0.0` |
| On-error replay enabled | ✅ `replaysOnErrorSampleRate: 1.0` |
| Sentry disabled in local dev | ✅ `enabled: process.env.NODE_ENV === "production"` |
| Build succeeds without auth token | ✅ `silent: true` |
| Enterprise tag taxonomy | ✅ `enterprise.*` tag prefix for all custom tags |

### 4.4 Source Maps

Source maps are uploaded automatically during Vercel builds when `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, and `SENTRY_PROJECT` are configured. Source map files are deleted from the deployment after upload — they are never exposed in the browser.

## Step 5: Resend Setup (Optional)

Resend handles transactional emails: welcome messages, password resets, and notifications.

### 5.1 Create Account and API Key

1. Go to [resend.com](https://resend.com) and sign up.
2. Navigate to **API Keys → Create API Key**.
3. Copy the key — it is shown only once.

### 5.2 Verify Your Domain

For production deliverability, verify your sending domain in **Resend → Domains → Add Domain**. Follow the DNS verification instructions. Without domain verification, emails may be sent from `onboarding@resend.dev` and may end up in spam.

### 5.3 Add to Vercel

Add to Vercel environment variables:

```
RESEND_API_KEY=re_...
EMAIL_FROM=noreply@yourdomain.com
```

## Step 6: CI/CD Pipeline

### CI Workflow (`ci.yml`)

The CI workflow runs on every pull request to `main` and every push to `main`. It is a **pure quality gate** — it does not deploy anything.

| Step | When | What it does |
|------|------|--------------|
| Typecheck | Always | `pnpm typecheck` — TypeScript strict check |
| Lint | Always | `pnpm lint` — Biome lint rules |
| Test | Always | `pnpm test` — Vitest unit tests |
| Build | Always | `pnpm build` — Next.js production build |

### Migration Deployment Workflow (`deploy-migrations.yml`)

Migration deployment runs in a **separate workflow** that is disabled by default.

| Step | What it does |
|------|--------------|
| Validate secrets | Fails with a clear message if required secrets are missing |
| Link project | `supabase link --project-ref $SUPABASE_PROJECT_ID` |
| Push migrations | `supabase db push` — applies pending migrations |

**To enable it:**

1. Add the repository variable `ENABLE_SUPABASE_MIGRATIONS=true` (see Step 2.3).
2. Add the three required secrets: `SUPABASE_ACCESS_TOKEN`, `SUPABASE_PROJECT_ID`, `SUPABASE_DB_PASSWORD` (see Step 2.2).

The workflow triggers on push to `main` and also supports `workflow_dispatch` for manual runs.

> **⚠️ Migrations are irreversible in production.** Always test migrations in a staging environment first. Use `supabase db diff` locally to preview what will be applied.

### Why Migration Deploy is Separate from CI

This is a **template repository**. When someone forks or clones it:

- The CI workflow must work immediately — typecheck, lint, test, build — without requiring any secrets.
- Migration deployment requires Supabase credentials that new users will not have yet.
- Coupling migration deploy into CI would cause failures on every push for unconfigured repos.
- Separating the workflows lets teams opt into automated deploys when they are ready, without touching the quality pipeline.

### Release Automation Note

This template uses [release-please](https://github.com/googleapis/release-please) (`.github/workflows/release.yml`) for versioning. It is important to understand what release-please does and does not do:

- **Does**: Creates and maintains a release PR that bumps versions based on conventional commits. When merged, it creates a GitHub release with a changelog and tags the commit.
- **Does not**: Add labels to your ordinary feature/fix PRs. It only manages its own release PR.
- **Does not**: Deploy anything. Deployment is handled by Vercel (auto-deploy on push to main) and the optional migration workflow.

You do not need to configure anything for release-please — it works out of the box with `GITHUB_TOKEN`.

## Step 7: Deploy

### 7.1 Push to Main

**Trigger production deploy**

```bash
git push origin main
```

Vercel auto-deploys every push to `main`. Monitor the build in the Vercel dashboard.

The CI pipeline will also run: typecheck → lint → test → build → deploy migrations.

### 7.2 Verify Production URL

After the deployment completes, visit your production URL and confirm:

- The landing page loads.
- `/sign-in` loads and accepts credentials.
- Auth redirects work (forgot password, sign-up confirmation).

## Post-Deploy Checklist

Run through this list after every first-time production deployment:

- [ ] Visit production URL — no 500 errors on landing page
- [ ] Sign in with a real user account
- [ ] Sign up a new user — verify tenant and profile are created
- [ ] Trigger a password reset — verify email arrives
- [ ] Access `/dashboard` — verify all data queries return correct tenant-scoped data
- [ ] Verify RLS: check that a user cannot access another tenant's data
- [ ] Open Sentry dashboard — confirm events are appearing after a test error
- [ ] Check Resend dashboard — confirm emails are being sent and delivered
- [ ] Run `supabase db push` — confirm no pending migrations

## Troubleshooting

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Build fails: "Cannot find module" | Missing workspace dependency | Check `turbo.json` build pipeline and workspace `package.json` |
| Auth redirect goes to wrong URL | `NEXT_PUBLIC_APP_URL` misconfigured | Set to exact production domain, including `https://` |
| RLS blocking all queries | Missing `tenant_id` in JWT `app_metadata` | Verify `handle_new_user()` trigger exists and runs on sign-up |
| "Localhost URL rejected in production" startup error | `NEXT_PUBLIC_APP_URL` set to `localhost` | Set to real production domain |
| Sentry not receiving events | Wrong DSN or `NEXT_PUBLIC_SENTRY_DSN` not set | Copy exact DSN from Sentry project settings |
| Source maps not uploaded | `SENTRY_AUTH_TOKEN` missing | Add token to Vercel env vars and GitHub Secrets |
| Emails not sending | `RESEND_API_KEY` missing or domain not verified | Add API key to Vercel env vars and verify sender domain |
| Storage uploads failing | Bucket not created or RLS policy missing | Create buckets manually and check `storage.objects` RLS policies |
| Vercel build uses wrong Node version | Missing `.nvmrc` or engines field | Add `"engines": { "node": ">=20" }` to root `package.json` |
| Migration deploy fails | `SUPABASE_ACCESS_TOKEN` or `SUPABASE_PROJECT_ID` missing | Add secrets to GitHub repo settings |
